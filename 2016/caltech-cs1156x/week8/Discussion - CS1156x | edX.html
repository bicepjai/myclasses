            say we have this matrix A=[2300007000]

            if we wanted to tacitly represent this as sparse in either language (pretend for the moment that Julia indexes at zero), by the following 3 arrays / vectors

            rowv=[01]

            colv=[02]

                        # R script to apply SVM with soft margin to data classification and use cross-validation for choosing the best model
library(e1071)
library(caret)

training <- read.table("features.train.txt", header=F)
colnames(training) <- c("dig","int","sym")
cost <- c(0.0001,0.001,0.01,0.1,1.0)

#1 versus 5
q = 2
training <- subset(training, training$dig == 1 | training$dig == 5)
training$dig <- ifelse(training$dig == 1,1,-1)

# SVM function
mySVM = function(fold) {
    model <- svm(as.matrix(fold[,2:3]),fold[,1],scale=F,type="C-classification",kernel="polynomial",degree=q,gamma=1,coef0=1,cost=c)
    fold$y <- predict(model, fold[,2:3])
		fold$match <- (fold$dig == fold$y)
    svsoft.miss = nrow(fold[fold$match == FALSE,])
    }

# 10 xFold validation runs
runs <- 100
r = 0 
miss = 0
results <- matrix(nrow = runs, ncol = length(cost))
while (r < runs) {
    r <- r + 1
    folds <- createFolds(training$dig)
    split_up <- lapply(folds, function(ind, dat) dat[ind,], dat = training)
    for (c in cost) {
        miss <- sum(unlist(lapply(split_up, mySVM)))
        ecv <- miss / nrow(training)
        results[r, log10(c) + 5] <- ecv
    }
}

winners <- c()
for (run in 1:runs) {
    winners <- append(winners, which.min(results[run,]))
}
print(table(winners))
print(colMeans(results))

                            2. TLDR: @ypeels was right, obviously.

                              Summary:

                              Doing more Monte Carlo iterations reduces not just the variance of people's results, but also the variance of the individual candidate models's results, and hence the variance of learning decisions based on their \(E_{CV}\).

                              It wouldn't make an individual model's \(E_{CV}\) a better estimate of \(E_{out}\) (that depends only on \(K=\frac{N}{10}\)), but it would make the result of comparing the different models (and the models of different people) based on their \(E_{CV}\) more reliable (by reducing the variance).

                              On the other hand, the fact that the \(C\) with the best \(E_{CV}\) actually has a worse \(E_{test}\) than the \(C\) with the worst \(E_{CV}\) (by ~14%, 0.0024 in absolute value) is food for thought, it makes me wonder whether doing too many iterations is really worth it.

                              Making our code deterministic with rng(1) doesn't, in general, really improve it's comparability to other people's code, or the comparability between models, it only helps when comparing the results between edits of our own code; so, it might be recommended during development.

                              And to clearly answer the original questions:

                              Is this to be expected?

                              Yes

                              And would a particular result be more correct than another or are they just different?

                              If both results are within the error bar, they are both probably correct even if different. If one of these results was obtained by doing more iterations, it will probably be more reliable (probably closer to the unknown \(\hat E_{CV}(g_C)\), the result of \(\infty\) iterations).

                            4. I think in the end the whole issue with the accuracy of \(E_{cv}\), at least for comparing different C's was not that important, as it could be worked-around using \(N_{mis}\) (the number of misclassified points) as metric instead. Of course given that the calculations were done over the whole 10 folds. The real issue here was the fact that the tie rule was used on top of statistical fluctuations. This introduced a bias between the C that is chosen most often and the C that has the best \(E_{cv}\) (see my experiment above with removing the bias).

                              Note, also that this bias introduces a consistent mismatch between the C chosen most often and the C with the better \(E_{cv}\), this cannot be solved by increasing accuracy and performing more runs. At least not in this experiment. We probably have to be careful not to draw big conclusions based on this experiment alone, as we would be overfitting this experiment. We probably need a lot more experiments to allow for good generalization of our conclusions :-).

                              But for now I think the best way to go is to keep track of all C's that have a minimum \(E_{cv}\) and not just to take the lowest C between them on each run. Only after all runs have been done, if there is still a tie between different C's then only once can you apply the tie rule and give preference to the lower C.

                              Although performing only 100 runs could be considered a bit low, perhaps after increasing it a bit perhaps to 250 or so, the best way in my opinion, is to spend more computer time on trying out many more C's which allows for drawing a graph. That would give much more insight into which (range of) C's give better performance. This may also allow for zooming in to certain regions of C's.
